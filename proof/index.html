Proof / read-only explainer

How Auditex stays read-only and keeps evidence local.

This page exists to answer the trust question directly. Auditex makes observed API use visible, records content-read status, fails unsafe bundles before handoff, and keeps raw evidence under operator control by default.

Still true: Auditex is not certification, not legal advice, and not guaranteed security. It is evidence tooling for human reviewers.

Public trust position

Read-only: audit and probe runs must not write to production tenants.

No content reads: public audit collectors must not read mail or file body content.

Local raw evidence: bundles stay where the operator writes them.

Reviewable proof: API inventory, proof tables, validation, blockers, and checksums travel with the pack.

Read-only proof chain

The trust story works only if the site shows each stage. That means concrete mechanics, not broad promises.

Probe first

Auditex checks auth path, profile fit, missing scopes, tenant blockers, licensing gaps, and local prerequisites before deeper collection.

Collect with explicit scope

Provider collectors gather tenant posture metadata needed for the selected audit surface and keep the observed call surface attributable to the run.

Record API inventory

Observed calls, permissions, read/write classification, and content-read status become a visible artifact inside the bundle.

Validate before handoff

Read-only and no-content-read checks flag unsafe or unsupported bundle states before customer-facing handoff.

Ship proof with the report

Report packs, proof tables, blockers, validation output, and checksums let the operator explain both findings and limits.

What public Auditex does not read

The strongest trust page says the forbidden reads plainly.

Surface Public audit position Reason to show it on the site
Gmail message bodies Must not read Removes fear that an audit tool silently inspects mailbox content.
Google Drive file content Must not read Keeps the product on metadata and sharing posture, not document content.
Exchange mailbox body content Must not read Makes mail posture evidence separate from message content.
SharePoint file content Must not read Shows SharePoint coverage is metadata and permission-focused.
OneDrive file content Must not read Same trust boundary for personal file stores.

What the operator gets back

Trust gets stronger when the output artifacts are visible before install.

API inventory

Observed call surface, permission context, read/write classification, and content-read status.

Proof tables

Customer-facing claims tied back to saved evidence and declared limitations.

Validation output

Unsafe bundle states, unsupported reads, and blocked checks made explicit before handoff.

Checksums and handoff packs

Package integrity for transfer and review.

Limits and blockers belong on the page

Auditex should not hide partial coverage behind scores or summary-only language.

Blocked checks

Missing scopes, tenant policy barriers, provider restrictions, and unsupported auth paths should stay visible in the output and in customer explanation.

Broad scopes with read-only posture

Some provider permissions have names that look broader than the public audit behavior. Auditex still records what was actually observed and whether content reads were attempted.

Human review still required

Findings, severity, business impact, and remediation decisions remain human work. Auditex packages the evidence, not the final professional judgement.
